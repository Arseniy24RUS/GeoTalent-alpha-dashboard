# Справка для построения дешбордов по STEM-данным

Три файла в `out/` (каждый — csv + parquet, одинаковое содержимое):

| Файл | Что внутри | Строк | Стран | Регионов |
|---|---|---|---|---|
| `STEM_STRICT_vacancies` | вакансии/постинги (flow, спрос сейчас) | 1552 | 40 | 481 |
| `STEM_STRICT_employment` | занятость (stock, кто уже работает) | 1243 | 37 | 182 |
| `STEM_companies_snapshot` | конкретные вакансии с названием компании (Adzuna) | ~10-15 тыс. (разовый снимок) | 8 | — |

**STRICT** во всех файлах означает "только core/professional-уровень, без техников" — правило применено единообразно по всем источникам, даже к тем, что не по нашей ISCO-таксономии (см. раздел "Источники" ниже).

## 1. Схема `STEM_STRICT_vacancies` / `STEM_STRICT_employment`

Обе таблицы — одна и та же схема (12 колонок), это единая факт-схема проекта (`common.FACT_COLUMNS`):

| Колонка | Тип | Смысл |
|---|---|---|
| `source` | string | eures / adzuna / bls / eurostat / eurostat_htec / statcan / uk_nomis |
| `metric` | string | vacancies / postings (только в vacancies-файле) · employment (только в employment-файле) |
| `occ_level` | string | всегда "occupation" в этих 2 файлах (industry-строки уже отфильтрованы) |
| `stem_group` | string | категория — **алфавит разный у разных источников**, см. таблицу ниже |
| `stem_class` | string | "core" у источников с нашей таксономией; `NaN` у HTEC/UK NOMIS (там свой критерий STRICT, не через эту колонку) |
| `occ_code_raw` | string | исходный код: `ISCO_GRP:ENG`, `SOC:15-1252`, `NOC:18`, `HTEC:HTC/OC2`, `SOC2010:200000` |
| `geo_country` | string/NaN | ISO alpha-2, ИЛИ `NaN` если строка — суб-региональная (см. пункт 2 ниже) |
| `geo_region` | string/NaN | **NaN = агрегат по стране; заполнено = суб-регион.** Ключевое правило, см. "Подводные камни" |
| `period` | string | формат РАЗНЫЙ по источникам: `2026-Q3`, `2026-07`, `2025`, `2025-Q2`, `2021-12` |
| `value` | float | само число |
| `unit` | string | `count` (штуки) / `thousands` (тыс. чел.) / `persons` (чел.) — **разные шкалы, не смешивать без пересчёта** |
| `snapshot` | string (ISO datetime) | когда именно был сделан запрос к API |

## 2. Схема `STEM_companies_snapshot`

8 колонок, одна строка = одна вакансия (не агрегат):

`country, region, stem_group, stem_class, company, title, created, snapshot`

`region` — второй уровень из `location.area` Adzuna (может быть `NaN`, если у вакансии не указан регион). `created` — дата публикации вакансии (не дата нашего сбора — та в `snapshot`).

## 4. Подводные камни (без этого дешборд будет лгать)

### 4.1 `geo_region` NULL/NOT NULL — это переключатель уровня агрегации, а не "есть/нет региона"
Строка с `geo_region=NaN` — это ИТОГ по всей стране. Строка с заполненным `geo_region` — это ЧАСТЬ той же страны. **Если сложить оба уровня по стране — получите задвоенную сумму.** Первый фильтр на любом дешборде: явно выбрать один уровень (`geo_region IS NULL` для карты по странам, `IS NOT NULL` для карты по регионам).

Исключение: у **Eurostat LFS** регионов вообще нет (только страны) — там этот вопрос не встаёт. У **BLS/StatCan/UK NOMIS** нац. агрегат (`geo_region IS NULL`) присутствует ОТДЕЛЬНО от регионов/штатов/провинций — тоже не складывать.

### 4.2 `unit` — три разные шкалы в одной колонке `value`
- `count` — штуки (вакансии/постинги, всегда целое)
- `thousands` — тысячи человек (LFS, HTEC, StatCan)
- `persons` — человек (BLS, UK NOMIS)

**Group by `unit` обязателен** перед любой суммой/сравнением `value`. Наглядный провал без этого фильтра: сложить BLS (persons, миллионы) с StatCan (thousands, десятки) — цифры несопоставимы на 3 порядка.

### 4.3 `stem_group` — три несовместимых алфавита
- **ENG / ICT / SCI** (наша ISCO/SOC-таксономия) — EURES, Adzuna, BLS, Eurostat LFS
- **HRST** (единый тег, без разбивки по дисциплинам) — Eurostat HTEC
- **UK_STEM_PROF** (единый тег) — UK NOMIS
- **NOC-производные ENG/ICT/SCI** у StatCan — *похожи по названию* на нашу таксономию, но это отдельная канадская классификация (NOC), не ISCO/SOC. Совпадение имён случайное, не смешивать напрямую как "тот же ENG".

Практически: на дешборде сравнение "ENG vs ENG" между странами валидно ТОЛЬКО между EURES/Adzuna/BLS/LFS. Канаду, EU-HTEC и UK показывать отдельными карточками/вкладками, не в одном ряду сравнения "по группе".

### 4.4 Разная свежесть данных — UK критично устарел
Большинство данных — 2023-2025. **UK NOMIS — данные за декабрь 2021**, таблица не обновлялась с 2022 (см. докстринг `fetch_uk_nomis.py`). Если дешборд покажет UK в одном ряду с остальными без пометки года — это будет читаться как актуальные данные, хотя это не так. **Обязательно** выводить `period`/год на дешборде рядом с любой цифрой по UK, а лучше — визуально помечать (другой цвет/значок "устарело").

### 4.5 Разный формат `period` — нельзя сортировать/парсить как один тип
`2026-Q3`, `2026-07`, `2025`, `2025-Q2`, `2021-12` — пять разных форматов. Для оси времени на дешборде нужно сначала привести к единой гранулярности (минимум — год, `period.str[:4]`), иначе сортировка/сравнение по датам будет ломаться.

### 4.6 Временных рядов почти нет — в основном одна точка
- **Vacancies-файл** — это ОДИН снимок (июль 2026), не история. Тренд по вакансиям в этих данных построить нельзя вообще.
- **Employment**: только Eurostat (LFS/HTEC) даёт 3 года (2023-2025) — там можно строить тренд. BLS/StatCan/UK — один снимок каждый, тренд не построить без повторного сбора в будущем.

### 4.7 EURES occstats (топ занятий) — не включён
В `STEM_STRICT_*` вошёл только режим `geo` у EURES. Режим `occstats` (occupation-без-региона) на момент сбора возвращал 0 STEM-строк (топ-18 занятий сайта временно без STEM) — не баг, но и не источник данных сейчас.

### 4.8 `STEM_companies_snapshot` — маленькая выборка, не перепись
Всего у Adzuna по этим 8 странам × 3 STEM-категориям — **~1.33 млн постингов**; в снимке — единицы тысяч (около 1%). Плюс в поле `company` попадают не только реальные работодатели, но и **рекрутинговые агентства/джоб-борды** (в топе встречались `OfferZen`, `JobRadars` — это площадки, не наниматели) и госструктуры целиком (`NSW Government`). Использовать как индикативную выборку, не как точный рейтинг работодателей. Файл — разовый (см. докстринг `oneoff_adzuna_companies.py`), повторный сбор в большом объёме упирается в ToS Adzuna.

## 5. Как начинать дешборд (рекомендованный порядок фильтров)

1. **Слайсер "уровень географии"**: страна vs регион (`geo_region` null/not null) — первым делом, до любых графиков.
2. **Слайсер/фильтр по `unit`** — обязателен перед любой суммой `value`.
3. **Разделить визуалы по `source`** (или хотя бы по группе "наша таксономия" vs "HRST/NOC/SOC2010") — не сравнивать `stem_group` через все источники в одной легенде.
4. Для UK — отдельная плитка/пометка с годом данных (2021).
5. Только после этого — карта (`geo_country`/`geo_region` + `value`), разбивка по `stem_group`, при наличии истории (Eurostat) — линейный график по `period`.

При соблюдении этого порядка сами визуалы (карта, барчарт по группам, тренд) строятся стандартными pivot-инструментами Power BI/Tableau/Superset за минуты — данные уже в long-формате, готовы к `GROUP BY`.
